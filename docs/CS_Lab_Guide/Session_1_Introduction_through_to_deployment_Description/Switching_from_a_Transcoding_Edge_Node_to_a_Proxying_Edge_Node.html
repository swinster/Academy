Switching from a Transcoding Edge Node to a Proxying Edge Node

No specific lab sheet section.

You have deployed Conferencing Nodes in the Transcoding role within the LAN and the DMZ. Nodes in the transcoding role decode incoming media and encode outgoing media, and any media sent between different transcoding locations use an automatic backplane. The backplanes are an essential concept in the world of Infinity as they carry media between the nodes used to host the same conference but distributed across different locations. Further, only media required in other locations is sent across the backplane. It is, therefore, far more efficient on bandwidth if all users connect to a local location. As a result, multiple transcoding locations are often deployed across geographically dispersed sites and data centres.

Deployment architectures such as geographically distributed systems or a Service Provider model allow users to join the transcoding location closest to them, reducing the last hop media path where the network connection may be unregulated. Infinity then automatically determines if users in other locations have connected to the same conference service, linking these locations using a backplane over a well-defined network connection.

As transcoding is compute-intensive, it might be more efficient to concentrate the high compute resources in one location (the LAN) for this traditional enterprise scenario. We can change the Infinity node in the DMZ to “proxy” media toward the LAN location.

Previously, we outlined that Infinity is a fully transcoding environment today, so the proxying edge role is all about getting media from a remote device to an Infinity transcoding node in another location. Therefore, it is essential to understand that the Proxying Edge role is not about proxying one remote users’ media directly to another; instead, it MUST be directed initially to a transcoding location.

Before making and change, run a test connection into a VMR using a single endpoint connecting to the Edge node. Note how things look in Live View and the conference graph.

To switch to a proxying solution, you need to complete two steps:

  • The node's role needs to be changed (from a Transcoding Conferencing Node to a Proxying Edge Node).
  • Then the transcoding resource of the location needs to be updated (to target a location that contains Transcoding Nodes. For example, the current transcoding target of "This Location" now only includes Proxy node(s), so it will be unable to perform any transcoding by definition.

That's it!

Please refer to https://docs.pexip.com/admin/edge_guidelines.htm.

You can convert any node and location very simply as the underlying software for each role type is identical. However, each node role has very different compute and host requirements. You should plan for these changes, understand what you might need to alter on the guest VM (such as vCPUs and vRAM within the hypervisor), and understand the knock-on effects on the host. For example, proxying nodes still require dedicated host resources. We are still dealing with real-time media, and overcommitting host resources will lead to poor call quality and performance. Also, be aware that you cannot mix and match different Conferencing Node role types (Transcoding and Proxying Edge) in the same System Location. Please refer to https://docs.pexip.com/admin/admin_planning.htm and https://docs.pexip.com/admin/server_design.htm.
