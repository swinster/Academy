Infinity Connect Client Registration and Gateway P2P calls

Suggested completion time: 40 minutes

More information

Use section #4.2 of the lab sheet.

To register a device to Infinity:

Configure Infinity to accept the registration:

  1. Setup a Device Alias, click Users & Devices --> Device Aliases, then click Add Device Alias.
  2. Add the following Information:
    1. Device alias (mandatory): add an alias that would usually look like a URI, for example:
       <name>@<Pexip infinity domain>
       
             Ensure it does not clash with any other alias already configured on your system.
    2. Enable Infinity Connect registration (non-SSO): Checked.
    3. If you would like other devices using alternative protocols to register to Infinity, then select as appropriate. NOTE: if you would like an H.323 device to register, you will likely need to configure two aliases per device; the H.323 ID and E.164 number. BOTH these aliases need to match what is configured on the endpoint to allow the device to register successfully.
    4. Add a username and password. For consistency, the username is often the same as the alias of the device (although could be just the user part, for example).
  3. Click Save.

 

Configure the Connect client to register to Infinity:

  1. Download and install the Connect client for Infinity on your computer (not the RDP server, and not the MMV client used by the Pexip Service).
  2. Once installed, open the Settings (click the three dots ···), then click Registrations.
  3. Registration Host: <Pexip infinity domain> (this works due to PexApp SRV records inserted in the DNS namespace that resolve to your Edge node, and of course you could have multiple Edge nodes for redundancy, so multiple SRVs).
  4. Alias, username, and password: configured as above in Infinity.

If you would like to register other devices to Pexip, you can use the same alias or create other aliases as you wish.

 

To allow calls to registered devices (where options are not listed below, defaults given in the rule should be used):

  1. Add a Routing Rule, click Services --> Call Routing, then click Add Call Routing Rule.
  2. Add the following Information:
    1. Name: Calls toward Registered Devices.
    2. Priority: 100.
    3. Outgoing calls from a conference: Checked (this allows devices to be called directly from a VMR).
    4. Destination alias regex match: .+@<Pexip infinity domain>
      (don’t forget to escape any periods in your domain in the RegEx pattern, e.g. .+@example\.com )
    5. Call target: Registered devices only.
  3. Click Save.

 

To dial from registered devices and VMRs to any external H.323 devices (where options are not listed below, defaults given in the rule should be used):

  1. Add a Routing Rule, click Services --> Call Routing, then click Add Call Routing Rule.
  2. Add the following Information:
    1. Name: Calls from registered devices and VMRs to external H.323 systems via the DMZ.
    2. Priority: 190.
    3. Outgoing calls from a conference: Checked (this allows devices to be called directly from a VMR).
    4. Match incoming calls from registered devices only: Checked.
    5. Destination alias regex match:
      ^([1-9][0-9]?|1[0-9]{2}|2[01][0-9]|22[0-3])(\.([0-9]|[1-9][0-9]|1[0-9]{2}|2[0-4][0-9]|25[0-5])){3}$
    6. Call target: Registered devices or external systems.
    7. Outgoing location: DMZ.
    8. Protocol: H.323.
  3. Click Save.

 

To dial from registered devices and VMRs, to any external SIP devices (where options are not listed below, defaults given in the rule should be used):

  1. Add a Routing Rule, click Services --> Call Routing, then click Add Call Routing Rule.
  2. Add the following Information:
    1. Name: Calls from registered devices and VMRs to external SIP systems via the DMZ.
    2. Priority: 200.
    3. Outgoing calls from a conference: Checked (this allows devices to be called directly from a VMR).
    4. Match incoming calls from registered devices only: Checked.
    5. Destination alias regex match: (?!.*@<Pexip infinity domain>$).*
    6. Call target: Registered devices or external systems.
    7. Outgoing location: DMZ.
  3. Click Save.

 

Testing

Before you test, ensure you have waited 1 minute for the configuration changes to synchronise across all the nodes. To test, create call flows that match each of the Routing Rules you have configured. HINT, when a Routing Rule is matched, a Gateway service instance is created, and the name of the service will take on the name of the Routing Rule plus an appended Globally Unique Identifier (GUID). This ensures that each gateway instance can be identified from the other. You can see this in the screenshots below.

For outbound SIP and H.323 test sites, try testing against the trainier deployment, VC20.

  • SIP – sip:loopback@ciscotac.net (or sip:loopback1@ciscotac.net, or sip:loopback2@ciscotac.net, or sip:loopback3@ciscotac.net).
  • H.323 IP dialing - 71.14.2.157
  • NOTE - sometimes, these do NOT work as expected, check out the blog post below. 

Also check out https://community.cisco.com/t5/collaboration-voice-and-video/loopback-testing-for-telepresence/ta-p/3156607 (beware, some addresses may no longer work, although during the previous testing, sip:loopback@ciscotac.net, 64.102.241.236, and 71.14.2.157 worked).
