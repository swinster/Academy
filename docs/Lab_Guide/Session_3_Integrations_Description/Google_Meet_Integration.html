Google Meet Integration

Use section #5 and section #11 of the lab sheet.

Conceptually, integrations with third-party systems are all similar in Pexip Infinity, especially when looking at the Infinity configuration. This Infinity configuration revolves around the need to set up Routing Rules (in a similar way to that which was described in the previous session relating to Gateway Point-2-Point calling). Of course, setting up the remote systems can be very different for different vendors, so it’s good to have an appreciation of how different vendors have designed their systems. Here, you will look at how Pexip integrates with Google Meet. 

Currently, Google Meet is a meet only platform. Participants dial into the Google Meet meeting; they can’t dial another endpoint directly. This behaviour may change in the future. We are therefore using Infinity to gateway standards-based devices into the Google Meet environment.

  1. The Incoming call leg to Infinity (the ‘A’ leg) can be from any device that cannot directly connect to Google Meet.
  2. The outgoing leg from Infinity (the ‘B’ leg) to Google uses HTTP(S) requests to POST messages or GET information from the Google side. 

 

Google side

The Google Meet settings have been configured for you already in Google Workspace, this part of the integration would normally be carried out by a Google Workspace user with Admin permission to make changes to the Google Meet configuration. There are two items which have been configured for your group:

  1. Gateway Interoperability: Gateway interoperability for your group (or Organisational Unit) has been enabled, which involves telling Google which remote host will be used for external VC endpoints (i.e. your Edge node IP address), the domain that will be used to construct the dialling instructions, and a numeric prefix that will be used to match call routing.

    2. NOTE: Google is moving to use the “long meeting IDs” in the future, hence the “preferred option” label. This gives a 13-digit code with no prefix, which will require a slightly different RegEx Rule match. Currently, for this lab, we have chosen to use both the long and short ID, although only the short ID seems to show in the HTML page for third party system dialling (which you will see in the coming steps). The short ID allows for a prefix to be added which can aid with call routing, plus help in controlling unsolicited callers where a generic 13 digit number may easily match a SPAM caller’s attempt.

    NOTE: The “additional gateway for Skype for Business users” is used when configuring integration with the Pexip Service, so not required for Infinity integrations.

  2. Interoperability Tokens: Two tokens are required; one for trusted devices, and one for untrusted devices. Google can generate one or more access tokens, which has been already done for you, and should be added to Pexip Infinity in the next step.

Please refer to https://docs.pexip.com/admin/gmeet_gsuite.htm.

 

Pexip Infinity side

Google Meet integration is an additional feature in Pexip and will require an additional license. You need one license for each access token you add, so for connections from trusted and untrusted sources, you will require two licenses. You will need to add the tokens generated in the previous step into Pexip via the Call Control menu.

You will also need to add two different call routing rules to allow the trusted and untrusted devices to connect. 

  1. Devices that are registered to your Pexip deployment should be treated as “Trusted” when connecting to Google Meet and will then be admitted directly into the Meet meeting. 

    2. NOTE: For v20+, there is a new Routing Rule option to allow a call matching the Rule to be treated as trusted regardless of being registered to Pexip; see https://docs.pexip.com/admin/gmeet_configuration.htm.

  2. Devices that are NOT registered to your Pexip deployment (such as external devices used by customers or business partners) will be treated as “Untrusted” when connecting to Google Meet. Only authenticated participants in the Meet meeting will be able to admit the call manually from a supported browser.

In the Routing Rule, you need to think about the outgoing call placement, and specifically the Outgoing Location for the ‘B’ leg of the call to the Google Meet cloud service. Should this be via the DMZ or LAN? 

Optionally, you can also configure a VR for lobby access dialling into Meet as described here:

https://docs.pexip.com/admin/gmeet_configuration.htm#ivr_route.
